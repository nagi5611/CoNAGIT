## 1. 本書の位置づけ

- マスター計画の **ストレージ・アップロード・サムネ／キュー** に含まれる要件のうち、**画像／動画の取り込み、クライアント側サムネ、外部変換、R2 保存、処理状態 DB** にスコープを絞る。
- **GIF は対象外**（JPEG / PNG / WebP のみ）。要件・UI・バリデータから除外する。

---

## 2. ゴール（本書で「完了」とする範囲）

| 領域 | 完了の定義（要約） |
|------|---------------------|
| 画像 | JPEG / PNG / WebP のみ。クライアントから **Presigned PUT** で S3 raw へ直接アップロード。 |
| 動画 | 原本は **クライアント直 S3**。先頭フレーム相当の静止画サムネはクライアントで生成し、**別オブジェクト**として S3 に PUT。 |
| サムネ生成（クライアント） | **第 1 手段: Mediabunny**（WebCodecs・遅延読み／ストリーミング I/O）。**`canDecode()` が false または失敗時のみ ffmpeg.wasm**。フォールバックは読み込み量・メモリ・時間が増えやすい旨を UI・仕様に明記。 |
| 変換〜配信 | Cloudflare 側で画像をリサイズ等し、得られたバイト列を **R2 公開バケット＋カスタムドメイン** 上の規約キーに保存。プロダクト選定は ADR 0006。 |
| メタデータ | 処理状態は **ローカル DB**（SQLite + Litestream 等）。**正規化した S3 オブジェクトキー**でファイルと突き合わせる。D1 との分担は ADR 0007。 |
| 負荷 | 大容量バイナリは **自宅／低性能サーバーを経由しない**。サーバーは Presigned 発行、キュー消費、軽量検証、外部 API オーケストレーション、DB 更新に限定。 |

**Mediabunny 出典（実装・契約前に再確認すること）**: npm パッケージ `mediabunny`、[Mediabunny Quick start](https://mediabunny.dev/guide/quick-start)。ライセンスは npm 表記（MPL-2.0）を確認のうえ利用する。

## 4. AWS 側の推奨構成

- **S3 → SQS**（**DLQ** 付き）。自宅サーバーの HTTPS 常時受信より **長期ポーリング** の方が動的 IP・回線品質に強い。
- **キー設計（例）**: `uploads/raw/{uploadId}/video.bin` と `uploads/raw/{uploadId}/thumb.jpg` のように **同一 `uploadId` プレフィックス**でペアリング。DB は処理対象を thumb 側に紐付ける／両方を行として持つ、のいずれかを実装前に **1 つに固定**する。
- **検証 NG**: 方針は **オブジェクト削除**。削除失敗時はリトライと手動掃除（運用 Runbook に記載）。

---

## 5. クライアント（`web/`）方針

- **Presigned 二段**（動画 raw + サムネ画像）: API から URL 取得 → ブラウザから S3 へ PUT。本番の API オリジンは `web/.env.production` と整合させる。
- **Mediabunny**: `Input` → `getPrimaryVideoTrack()` → **`canDecode()` が true** のとき、`CanvasSink` または `VideoSampleSink` で先頭付近の 1 枚を Canvas に描画し、**JPEG または WebP の Blob** として thumb キーへ PUT。
- **ffmpeg.wasm フォールバック**: **ファイルサイズ上限**、**進捗表示**、**タイムアウト**を必須とする。MVP では **MP4 + H.264 のみ** 等に限定する選択肢を検討可能。
- **画像のみ**: raw を S3 に PUT。サムネを常に別キーに書くかは実装ゲートで固定する。

### サムネ失敗時のポリシー（実装で 1 行に固定）

1. Mediabunny を試行し、成功時のみ thumb を PUT。  
2. 失敗時のみ ffmpeg.wasm。  
3. いずれも失敗した場合は **「動画のみ許可」** か **「全体エラー」** のどちらかを採択し、UI と API のエラーコードを一致させる。

## 6. サーバー（Node / API）の責務

- 画像デコードの本格処理は **必須としない**（最小: magic byte、`Content-Type`、サイズ上限）。厳密検査を入れる場合は別ワーカー化と頻度制限を文書化する。
- 既存コードの接続点（リポジトリ上の想定パス）: `src/routes/upload.ts`、`src/thumbnail/enqueue.ts`、`src/thumbnail/process-queue-message.ts`。キューが Cloudflare Queues 前提の場合は **SQS アダプタ**で同一インターフェースに差し替える方針とする。

---

## 7. DB とバックアップ

- **第一推奨**: **SQLite + Litestream**（レプリカ先は **R2 または S3 互換**）。Docker 不要。  
- **同時書き込み**: キュー消費は原則 **シングルワーカー**、または WAL + 短トランザクション。  
- **代替**: OS パッケージの PostgreSQL + `pg_dump` 定期退避（リソースは SQLite より重い）。  
- **バックアップ・リストア**: [Litestream Runbook](./runbooks/litestream-backup-restore.md)、公式 [S3-compatible ガイド](https://litestream.io/guides/s3-compatible/)。

---

## 8. R2・配信

- **パブリック読取 + カスタムドメイン**（要件確定済み）。公開キー規約の例: `optimized/{uploadId}/{variant}.webp`。キャッシュヘッダ、誤公開時の緊急手順は既存 Runbook（例: `docs/runbooks/aws-s3-cloudfront.md`）とリンクする。

## 9. 状態遷移（案）

| 状態 | 意味 |
|------|------|
| `received` | S3 にオブジェクトが存在し、キューに通知済みまたは検知済み |
| `validating` | HEAD / 範囲 GET 等で軽量検証中 |
| `transforming` | Cloudflare 画像変換の取得中 |
| `r2_written` | R2 に最適化オブジェクトを書き込み済み |
| `failed` | 失敗（DLQ または再試行ポリシーに従う） |

## 10. マイルストーン（実装順）

1. キー規約・DB スキーマ案の確定（S3 キー正規化、上記状態遷移）。→ **コード**: `migrations/0005`・`0006`、`MediaPipelineDb`、`validate-s3-thumb`  
2. Presigned 二段（動画 + サムネ）の API とクライアント PoC（Mediabunny）。→ **コード**: `media-presign-pair`・`web/` PoC・動画のみフォールバック  
3. S3 通知 → SQS → プロセッサ。→ **コード**: `media-pipeline-sqs-worker`（SQS 長期ポーリング）※AWS コンソール配線は本番作業  
4. Cloudflare 変換 → R2 PUT（ADR 0006 に合わせた実装）。→ **コード**: ワーカー内 `fetch` + `MEDIA_CF_TRANSFORMS_URL_TEMPLATE`  
5. 監視・DLQ・バックアップ復元テスト。→ **コード**: CF リトライ・失敗時 DB `failed`／**運用**: DLQ・Litestream は Runbook 側  
6. E2E（`e2e/` にクリティカルパスを追加）。→ **コード**: `e2e/media-ui.spec.ts` + `npm run e2e`

## 14. フェーズ2（着手済み）— キー規約・DB・クライアント PoC

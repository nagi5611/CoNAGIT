# Runbook: Litestream（SQLite レプリカ）バックアップとリストア

| 項目 | 内容 |
|------|------|
| 対象 | メディア処理パイプライン用 **ローカル SQLite** の継続バックアップ（計画: ADR 0007） |
| 前提 | Litestream バイナリがプロセッサホストにインストール済み。レプリカ先は **S3 互換**（AWS S3 / Cloudflare R2 等）。 |

## 1. 公式ドキュメント（正）

- S3 互換ストレージへのレプリケーション（**R2 のエンドポイント例が表に含まれる**）: [https://litestream.io/guides/s3-compatible/](https://litestream.io/guides/s3-compatible/)  
- リストア CLI（バージョンに応じたサブコマンド）: 上記サイト内の **Restore** ドキュメントおよび `litestream restore -h` のヘルプを参照する。

## 2. 設定チェックリスト（運用開始前）

- [ ] `replica.url` または `type: s3` + `endpoint` が対象プロバイダ（例: `*.r2.cloudflarestorage.com`）と一致している。  
- [ ] `access-key-id` / `secret-access-key` は **環境変数またはシークレット管理**から注入し、リポジトリに含めない。  
- [ ] レプリカ先バケットの **ライフサイクル・バージョニング**方針が組織ルールと一致している。  
- [ ] Litestream の **メジャーバージョン**をサーバーと CI のドキュメントに固定し、破壊的変更時は再検証する。

## 3. バックアップが正常かの確認

- レプリカ先に **世代が増えている**こと（オブジェクト一覧でプレフィックスが更新される）。  
- Litestream のログに **エラーなく snapshot / WAL** が進んでいること。  
- 月次: テスト用 DB で **リストア手順のリハーサル**を実施し、手順書のコマンドを更新する。

## 4. 障害時（ローカル DB 破損）

1. アプリ／プロセッサを **停止**し、残っている WAL を可能な限り保全する。  
2. 公式手順に従い、直近のレプリカから **`litestream restore`** を実行する（出力先パスは本番パスと別に検証してから切替）。  
3. リストア後に **整合性**（行数、最新 `updated_at`、S3 キー一意制約）を確認してからプロセスを再起動。  
4. 原因がディスク満杯・権限失効・ネットワーク遮断のいずれかを **インシデント記録**に残す。

## 5. シークレット失効時

- ストレージ側でキーをローテーションしたら、**Litestream 設定を先に更新**し、再起動後に旧キーを失効させる（順序誤りでレプリケーションが止まる）。
